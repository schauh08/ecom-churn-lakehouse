# ADR-0001: Use Delta Lake for Bronze, Silver, and Gold

- **Status:** Accepted
- **Date:** 2026-05-07

## Context

The project needs:

- append-only Bronze ingestion
- idempotent Silver trusted publishes
- reproducible Gold feature snapshots
- safe reruns and backfills
- metadata-rich lineage for training and serving

Plain Parquet alone is not enough for safe merges and trusted publish semantics.

## Decision

Use **Delta Lake** tables for Bronze, Silver, and Gold.

Layer-specific behavior:

- **Bronze:** append-only raw evidence plus ingest metadata
- **Silver:** ACID-backed merge for trusted publish and deterministic dedupe
- **Gold:** reproducible point-in-time snapshots with snapshot metadata

## Consequences

### Positive
- ACID semantics for trusted publishes
- safer reruns and backfills
- consistent Spark-native storage and read/write path
- simpler contract enforcement at Medallion boundaries

### Negative
- Spark + Delta adds more local setup than a pure pandas/parquet project
- local development depends on Java and PySpark compatibility

## Rejected alternatives

### Plain Parquet only
Rejected because it does not give safe merge semantics for Silver.

### Database-first storage
Rejected for V2 because it would add operational surface area without improving the thin-slice demonstration.
