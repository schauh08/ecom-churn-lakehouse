# ADR-0002: Enforce a fail-fast quality gate at the Silver boundary

- **Status:** Accepted
- **Date:** 2026-05-07

## Context

Silver is the trusted boundary in this repo. If invalid records silently publish to Silver, every downstream layer becomes less trustworthy:

- Gold features become incorrect
- labels can be wrong
- training snapshots can drift
- API outputs can be hard to defend

## Decision

Treat the Silver DQ gate as **blocking** for critical checks.

Current blocking checks:

- `order_id` not null
- `customer_id` not null
- unique `order_id`
- allowed `order_status`

If a critical check fails:

- publish does not proceed
- a DQ report is written
- failed-row samples or quarantine outputs are preserved when available

## Consequences

### Positive
- trusted Silver really means trusted
- bad data is surfaced early
- downstream features and model artifacts are easier to defend
- backfills and reruns fail loudly instead of corrupting trusted tables

### Negative
- a dirty input batch can block the pipeline
- operators must inspect artifacts and rerun after remediation

## Rejected alternatives

### Warn-only quality checks
Rejected because the project’s thin slice is explicitly built around Silver as the trust boundary.

### Gold-only validation
Rejected because it moves failure detection too late in the flow.
