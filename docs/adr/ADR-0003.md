# ADR-0003: Serve predictions from a local latest-features export instead of recomputing features online

- **Status:** Accepted
- **Date:** 2026-05-07

## Context

The API needs to return real predictions with traceable `model_version` and `feature_version`, but this V2 build is intentionally lean and local-first.

Possible approaches were:

- recompute customer features online on every request
- build an online feature store
- export the latest Gold snapshot per customer and let the API read that local artifact

## Decision

Use a **local or offline latest-features export** built from Gold.

The serving layer materializes the most recent Gold snapshot per customer and stores it as parquet for the API to read.

The API:

- loads the approved model artifact locally
- loads the latest-features export locally
- fetches features for `customer_id`
- scores prediction without recomputing features at request time

## Consequences

### Positive
- simpler serving path for a portfolio project
- avoids training-serving skew caused by duplicating feature logic inside the API
- keeps latency and operational complexity low for V2
- supports version checks between model and feature export

### Negative
- serving freshness depends on rebuilding the latest-features export
- this is not a full online feature-store architecture

## Rejected alternatives

### Online feature recomputation in the API
Rejected because it duplicates feature logic and increases skew risk.

### Redis or Dynamo-backed online store
Rejected because it is out of scope for the lean V2 build.
